## 1. このツールの前提

- **目的**：各プールの **休館日という「事実」** を 1 日 1 回、公開ページから取得し、自前 UI のカレンダーで表示する。出典リンク＋「最終確認は公式で」の注記つき、**非商用**。
- **法的な立ち位置**：取得対象は *事実* であり著作権の対象外。相手ページの**文章・レイアウト・PDF ファイルそのものを転載しない**限り「複製」にあたらない。→ **事実を自分の形（DATA{}）で出す**方針を厳守。
- **レッドライン（やらない）**：
  - ログイン内側 / CAPTCHA 突破 / 高頻度アクセス
  - 相手コンテンツの丸写し（文章・HTML・PDF の転載）
  - 商用化
  - ※横浜国際等の「空き状況（予約システム）」は reCAPTCHA・セッション・ログイン圏のため**本ツールでは扱わない**（休館日という事実だけを扱う）。
- **礼儀**：
  - 1 日 1 回・1 ページ、アクセス間隔を空ける
  - User-Agent に正体と用途を明記（例：`pool-calendar updater (contact: ...)`）
  - robots.txt を尊重
  - noindex で検索非掲載

## 2. robots.txt の読み方

- 「robots.txt がある／ない」ではなく、**取りたいページのパスが、自分に適用される User-Agent の `Disallow` に入っているかだけ**を見る。
- 管理画面 / 検索パスの拒否は、休館日ページの取得とは**無関係**。
- `ClaudeBot` 等 AI ボットの**名指し拒否**がある場合は、一般 UA に適用されなくても *意思表示* として尊重を検討する。

## 3. 新プール追加前に必ずやること（Claude Code で実施）

1. **robots.txt を実取得**し、取得したいパスの `Disallow` を確認
2. **利用規約 / サイトポリシー**に「自動取得・転載・複製禁止」の記述がないか確認
3. **休館日の掲載形態**を確認（静的 HTML 表／文章／月次 PDF／年間 PDF／カレンダー／予約システム）

→ そのうえで [README 相当の追加手順]：① `index.html` の `POOLS` に定義追加 → ② `update_data.py` の `POOLS` に設定追加 → ③ 必要なら新しいソース型の parser を書いて `DISPATCH` に登録。

## 4. 13 プールの現況（★2026-07-01 実取得で検証済み）

> 下表は **実際に robots.txt とページを取得して確認した結果**。前回の「推定」から一部逆転している（特に京王アリーナ）。
> `➕ 着手可` / `⚠️ 条件付き` / `⛔ 回避推奨` の3区分。多くは**固定ルール（曜日ベース）でスクレイピング不要**な点が判明。

### ⚠️ 重要教訓：robots.txt に記述が無くても WAF が AI ボットを弾くことがある
robots.txt が無い/緩いことは「取得OK」を意味しない。**サーバ/WAF が `ClaudeBot`/`GPTBot` の User-Agent を 403 で遮断**している施設が実在する（京王アリーナ・町田で確認）。
新プール追加前に、休館日ページに対し **UA を変えて応答コードを実測**（例）すること：
```bash
for UA in "…ClaudeBot…" "…GPTBot…" "…Chrome…"; do
  curl -s -o /dev/null -w "%{http_code}\n" -A "$UA" "<休館日ページURL>"; done
```
AI ボット UA だけ 403 なら、一般 UA での取得は技術的に可能でも「先方の明示的拒否の回避」になるため**回避推奨**（許諾確認を要件化）。

**京王アリーナ TOKYO の実測（2026-07-01, このMacから直接）:**
| 対象 | ClaudeBot UA | GPTBot UA | 一般ブラウザ UA | 当スクレイパー UA |
|---|---|---|---|---|
| `/news/closed.php` | **403** | **403** | 200 | 200 |
| `/robots.txt` | **403** | — | 404（不在） | — |

### 代官山スポーツプラザ（渋谷区/ミズノ指定管理）

### 重要な着眼点：多くが「固定ルール」で取得不要
検証の結果、引地台・世田谷・相模原（定例）・東砂・都筑・代官山は **休館日が曜日ベースの固定ルール**（毎週月／第2・第4月／第3月／毎月最終火 等＋年末年始）で決まっており、既存の善行(zen)と同じ **`RULES` エンジンでカレンダー生成できる＝スクレイピング不要／低リスク**。臨時休館だけを軽く監視する構成が堅い。
